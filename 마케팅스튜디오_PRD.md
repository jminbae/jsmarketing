## 1. 개요

### 1.1 한 줄 요약

마케팅팀이 매일 반복하는 이미지·영상·문서·QR 작업과 **반복 릴스 제작**을, **외부 SaaS 구독 없이·파일 업로드 없이·사내 워크플로우에 맞춰** 처리하는 자체 제작 도구 모음이다.

### 1.2 배경 및 문제 정의

현재 마케팅팀은 QR 생성, 누끼따기, 이미지/영상 업스케일, GIF 제작, PDF 편집, 영상 자막, 반복 릴스 제작 등을 각각 다른 외부 서비스(Canva, remove.bg, iLovePDF 등)로 처리하고 있다. 이 방식에는 세 가지 구조적 문제가 있다.

1. **개인정보 위험** — 피부과 특성상 환자 사진·동의서를 다루는데, 외부 SaaS는 파일이 그쪽 서버로 전송된다. 개인정보보호법 측면에서 통제 불가능한 지점이 생긴다.
2. **비용·제약** — 무료 등급은 사용량 제한·워터마크·광고가 붙고, 유료는 매달 고정비가 나간다.
3. **워크플로우 불일치** — 브랜드 컬러·로고·자주 쓰는 규격이 도구마다 흩어져 있어 매번 수작업으로 맞춰야 하고, 흐름이 거의 같은 릴스도 매번 처음부터 만든다.

### 1.3 목표 (Goals)

- 자주 쓰는 작업을 **한 곳에서** 처리한다.
- **파일이 외부로 나가지 않는 구조**(클라이언트 처리 또는 로컬 처리)를 기본값으로 한다.
- **월 고정비 0원**에 수렴하는 운영 구조를 만든다.
- 맥·윈도우 혼합 팀에서 **동일하게** 동작하게 한다(크롬 기준).
- 결과물은 저장 없이 **즉시 다운로드**한다.
- **반복되는 릴스·콘텐츠를 템플릿으로 양산**해 제작 시간을 줄인다.

### 1.4 비목표 (Non-Goals)

- 결과물의 클라우드 저장·계정·협업 기능은 만들지 않는다(저장 불필요 요구사항 반영).
- 외부 사용자·환자 대상 공개 기능은 범위 밖이다(그건 피부텐텐의 역할).
- 클라우드 GPU를 빌려 쓰는 종량제 처리 방식은 채택하지 않는다.
- 영상 업스케일·영상 렌더를 "브라우저 안에서" 처리하는 시도는 하지 않는다(2트랙 구조에서 로컬로 분리).

## 2. 사용자

### 2.1 대상 사용자

사내 마케팅팀 구성원(비개발자 포함). 유튜브 채널(약 20만 구독) 콘텐츠 제작·SNS·블로그·피부텐텐 운영 인력.

### 2.2 페르소나

- **콘텐츠 에디터**: 영상 편집·자막·썸네일·GIF·릴스가 주 업무. 대용량 영상(1~2GB)을 자주 다루고, 흐름이 비슷한 릴스를 반복 제작한다.
- **SNS/블로그 담당**: 누끼·리사이즈·압축·QR·비포애프터 이미지가 주 업무. 빠르고 가벼운 도구가 필요하다.
- **운영/관리자(원장)**: 도구를 배포·관리하고, 시술명 마스터 사전과 릴스 템플릿을 유지한다.

### 2.3 사용 환경

- **OS**: 맥(애플실리콘 중심)과 윈도우 혼재.
- **브라우저**: 크롬 통일.
- **하드웨어**: **모든 PC에 GPU 탑재**(자막·업스케일·릴스 렌더 로컬 처리에 유리한 전제).

## 3. 핵심 설계 원칙 (아키텍처 철학)

본 제품의 모든 의사결정은 아래 원칙에서 파생된다.

1. **파일은 밖으로 나가지 않는다.** 처리는 사용자의 브라우저 안(트랙 1) 또는 사용자의 PC 안(트랙 2)에서 일어난다. 어떤 작업도 파일을 원격 처리 서버로 보내지 않는다.
2. **연산 주체는 사용자의 기기다.** 운영자가 무거운 처리 서버를 켜둘 필요가 없다. 따라서 처리 비용과 월 고정비가 0에 수렴한다.
3. **저장하지 않고 즉시 내려받는다.** 결과물은 메모리에서 생성되어 바로 다운로드된다. 서버 보관·DB 적재가 없다.
4. **공유가 필요한 것은 "데이터"뿐이다.** 영상·이미지 연산은 각자 로컬에서 하되, **시술명 사전·릴스 템플릿 같은 텍스트/규격 데이터만** 한곳에서 공유한다.
5. **하나만 고치면 양쪽이 따라온다.** 대중(피부텐텐)과 내부(자막툴)가 같은 시술명 원본을 바라보게 해, 업데이트를 1회로 끝낸다.
6. **한 번 설계, 무한 변주.** 반복 작업(릴스)은 템플릿으로 만들어 변수만 갈아끼운다.

## 4. 제품 구조 — 두 개의 트랙

| 구분 | 트랙 1: 브라우저 웹툴 | 트랙 2: 로컬 웹앱 |
|---|---|---|
| 설치 | 불필요 (URL 접속) | 최초 1회 설치 (각 PC) |
| 처리 위치 | 브라우저 메모리(WASM) | PC 네이티브 엔진(GPU 활용) |
| 적합 용량 | 작은 파일(수백 MB 이하) | 대용량(1~2GB+) |
| 화면 | 웹(배포 도메인) | 웹(localhost), 화면은 동일 경험 |
| 대표 기능 | QR·누끼·압축·리사이즈·PDF·짧은 GIF | 영상 자막·대용량 영상 처리·영상 업스케일·**릴스 템플릿 자동화** |
| OS 차이 | 없음(완전 동일) | UI는 동일, 설치법·속도만 상이 |
| 비용 | 호스팅 사실상 0원 | 엔진 무료 OSS, 처리 0원 |

**왜 둘로 나누는가**: 브라우저 샌드박스는 메모리 한계로 보통 500MB~1GB를 넘기면 불안정하다. 따라서 가벼운 작업은 트랙 1로 즉시 제공하고, 대용량 영상·AI 업스케일·릴스 렌더는 PC 자원을 직접 쓰는 트랙 2로 분리한다.

## 5. 트랙 1: 브라우저 웹툴 (상세 기능 명세)

> 공통 동작: 파일 드래그&드롭 또는 선택 → 브라우저 내 처리 → 결과 즉시 다운로드. 업로드·저장 없음. 맥/윈도우 크롬 동일.

### 5.1 기능 일람

| # | 기능 | 입력 | 출력 | 핵심 라이브러리 | 구현 난이도 |

### 5.2 주요 기능 상세

**① QR 코드 생성기**
점(dot) 모양·모서리 스타일·전경/배경 색·그라데이션·색 반전·로고(힐하우스/피부텐텐) 삽입을 지원한다. 자주 쓰는 브랜드 프리셋을 버튼화한다. SVG(인쇄용)·PNG(웹용) 즉시 다운로드.

**② 이미지 누끼**
브라우저에서 AI 모델(WebGPU)로 배경을 제거한다. 사진이 서버에 올라가지 않는 것이 가장 큰 장점이다. *제약*: 최초 모델 다운로드 약 40~80MB, 머리카락·반투명 경계 디테일은 전용 유료 서비스(remove.bg·포토샵)보다 한 수 아래다.

**③ 이미지 압축·포맷 변환**
용량 압축, WebP 변환(피부텐텐 SEO·로딩속도 개선에 직결), HEIC→JPG 변환(아이폰 사진 호환)을 처리한다.

**⑤ EXIF 제거 · ⑦ 모자이크/블러**
환자 사진을 다루는 피부과 특성을 반영한 **비식별화 묶음**이다. EXIF 제거는 촬영 위치·기기정보를 세탁하고, 모자이크/블러는 얼굴·특정 부위를 가린다. 외부 공유 전 개인정보 정리에 사용한다.

**⑥ 비포&애프터 슬라이더**
피부과 마케팅의 핵심 포맷. 두 이미지를 좌우 분할 비교로 합성한다. 결과는 이미지 또는 웹 임베드 형태로 내보낸다. **영상 버전은 트랙 2의 릴스 템플릿 자동화(§6.6)로 확장된다.**

**⑨ 짧은 영상 → GIF**
`ffmpeg.wasm`(FFmpeg을 브라우저로 컴파일)으로 자르기·리사이즈·fps/팔레트 최적화를 처리한다. SNS용 짧은 클립에 적합하다. *제약*: 최초 약 25MB 로드, 긴 영상은 메모리 한계로 부적합(→ 트랙 2).

**⑩ 작은 이미지 업스케일**
`UpscalerJS`(Real-ESRGAN 계열)로 썸네일·로고 등 작은 이미지를 확대한다. *제약*: 큰 이미지는 메모리·속도 한계로 수십 초~분 단위. WebGPU 환경에서 가속된다. 대용량은 트랙 2로 보낸다.

### 5.4 배포 요구사항

- 정적 호스팅(Vercel / Netlify / Cloudflare Pages) 또는 기존 Next.js 스택에 페이지로 편입.

## 6. 트랙 2: 로컬 웹앱 (상세 기능 명세)

> 화면은 트랙 1과 같은 웹(localhost) 경험이지만, 무거운 연산은 PC에 설치된 네이티브 엔진이 담당한다. 1~2GB 영상도 GPU를 활용해 처리한다.

### 6.1 동작 방식

1. 최초 1회 설치(설치 스크립트, 약 10~20분 / 클릭형 제공 목표).
2. 이후 바탕화면 아이콘 더블클릭 → 로컬 서버 기동 → 브라우저 자동 실행.
3. 사용자는 웹 화면에서 작업, 엔진(ffmpeg·Whisper·Real-ESRGAN)이 로컬에서 처리.
4. 결과물 즉시 다운로드(로컬 저장 위치 지정).

### 6.2 기능 일람

| # | 기능 | 엔진 | GPU 활용 | 비고 |
|---|---|---|---|---|
| 1 | 영상 자막 생성·입히기 | Whisper + ffmpeg | ✅ | §6.3 |
| 2 | 대용량 영상 변환/자르기/리사이즈 | ffmpeg(네이티브) | ✅ | 1~2GB+ 가능 |
| 3 | 영상 업스케일 | Real-ESRGAN / Video2X / Topaz | ✅ | §6.4 |
| 4 | 릴스 템플릿 자동화(+배치) | ffmpeg / Remotion | △~✅ | §6.6 |
| 5 | (확장) 영상 → 고품질 GIF | ffmpeg(네이티브) | △ | 긴 영상도 처리 |

### 6.3 영상 자막 (핵심 기능)

**처리 흐름**

```
[영상] → Whisper(음성인식, 타임코드 포함 SRT 생성)
       → LLM(전문용어·오타 교정, 필요시 번역) ← 공유 시술명 사전 참조
       → ffmpeg(자막 굽기 hard-sub / 트랙 첨부 soft-sub)
       → [자막 완성 영상] 즉시 다운로드
```

**중요 개념**: 자막 생성은 LLM이 아니라 **음성인식(ASR) 모델인 Whisper**가 수행한다. Whisper는 "소리→글자" 전용 모델이며 **완전 무료 오픈소스(MIT 라이선스)** 다. LLM은 2단계 교정·번역에만 사용한다.

**정확성**: 깨끗한 한국어 음성 기준 90~95% 수준. 단, 전문용어(F-VASI, 스킨부스터, 리프팅 기기명 등)는 Whisper가 자주 틀린다. **사람 검수 1회는 필수**이며, 공유 사전 기반 LLM 교정으로 상당 부분 자동 보정한다.

**모델 권장**: `whisper.cpp` 또는 `faster-whisper`. 애플실리콘에 최적화되어 있어 **맥(M칩)에서 특히 빠르고 정확**하다.

### 6.4 영상 업스케일

- **무료**: Real-ESRGAN, Video2X.
- **상용(고품질)**: Topaz Video AI(전량 로컬, GPU 필요). 가격은 §10 참조.
- 전제: 모든 PC에 GPU가 있으므로 각 PC에서 자체 처리한다. NVIDIA GPU에서 가장 빠르며, 1~2GB 영상도 시간을 들이면 처리 가능하다.
- **권장 순서**: 무료 Real-ESRGAN으로 먼저 화질을 검증하고, 부족할 때만 Topaz 도입을 검토한다.

### 6.5 배포·설치 정책

- **각 PC에 설치**(모든 PC에 GPU가 있으므로, 한 PC 집중 대신 분산 처리). 작업이 몰리지 않고 각자 자기 GPU로 빠르게 처리한다.
- 설치 스크립트는 **맥용 / 윈도우용**을 별도 제공한다. **화면(UI)은 양쪽 동일**, 차이는 설치 절차와 처리 속도뿐이다.

### 6.6 릴스 템플릿 자동화 (반복 콘텐츠 양산)

**개념**
릴스의 흐름·음악·전환·자막 위치는 **고정 템플릿**으로 두고, 매 편 바뀌는 요소만 **변수 슬롯**으로 교체한다. 흐름이 거의 동일하고 전/후 사진과 일부 텍스트만 달라지는 반복 제작에 최적화된 자동화다.

**변수 슬롯 (예시)**

| 슬롯 | 내용 |
|---|---|
| `before_image` | 시술 전 사진 |
| `after_image` | 시술 후 사진 |
| `age` | 나이 |
| `gender` | 성별 |
| `comment_text` | 코멘트/설명 자막 |
| (확장) | 시술명, 자막 컬러, 인트로 문구 등 |

**작업 흐름**
1. 템플릿 선택
2. 전 사진·후 사진 드래그&드롭
3. 나이·성별·코멘트 입력
4. 렌더 → **9:16 세로 영상** 즉시 다운로드

**배치(일괄) 생성**
표(구글시트/엑셀)에 케이스 N건을 적고 사진을 폴더에 넣으면 → 릴스 N편을 한 번에 렌더. 대량 운영에 최적이다.

**위치**: 트랙 2(로컬). 음악·전환이 들어간 영상 렌더는 브라우저엔 무겁고, 무워터마크·고화질·고속을 위해 로컬 GPU를 쓴다. 화면은 웹, 렌더는 PC가 담당한다.

**빌드 방식 (두 갈래)**

| 방식 | 특징 | 비용 |
|---|---|---|
| ffmpeg 파이프라인 | 흐름 고정형, 변수만 교체. 단순·견고 | 무료 |
| Remotion | 전환·애니메이션 유연(React 기반 코드 템플릿) | 회사 규모에 따라 상업용 라이선스 비용 가능 → 도입 전 확인 |

**선행 조건**: 정확한 템플릿 설계를 위해 **실제 릴스 1편**(또는 흐름 명세 — 인트로 길이·전환 방식·자막 위치·음악 유무)이 필요하다.

**연관**: 트랙 1 비포·애프터 슬라이더(§5.2 ⑥)의 영상 확장판이다.

## 7. 공유 사전 아키텍처

대중용 피부텐텐과 내부 자막툴이 시술명을 **한 원본**에서 공유하되, 성격이 다른 데이터는 분리한다.

### 7.1 두 사전의 성격 구분

| 구분 | 내부 교정용 사전 | 피부텐텐 시술 사전 |
|---|---|---|
| 목적 | 음성인식 오류 교정 | 환자 대상 정보 제공 |
| 예시 | `에프바시 → F-VASI`, `울쎄라 → Ultherapy` | 시술 설명·태그 분류(6개 카테고리) |
| 노출 | 내부 전용 | 대중 공개 |

### 7.2 단일 원천(Single Source of Truth) 구조

```
        [표준 시술명 마스터]  ← 원장이 여기만 관리
         procedures_master (Supabase)
              │  (읽기 전용, 한 방향)
      ┌───────┴───────┐
      ▼               ▼
 [피부텐텐]        [내부 자막툴]
 대중용 설명·태그   교정용 별칭 매핑
 로 확장해서 사용   (asr_aliases, 내부 전용)
```

- 가운데 **표준 시술명 마스터**가 공식 표기·분류의 원본이다.
- 새 기기/시술을 마스터에 1회 추가하면 → 피부텐텐(시술 카드)·내부 툴(교정 사전)이 **동시에 최신화**된다.

### 7.3 데이터 모델(초안)

| 테이블 | 위치 | 주요 필드 | 공유 |
|---|---|---|---|
| `procedures_master` | Supabase | 공식명, 카테고리(6종), 영문표기 | 양쪽 공유(원본) |
| `asr_aliases` | Supabase | 발음 변형 → master 참조 | 내부 전용 |

> 6개 카테고리: 리프팅 · 스킨부스터 · 필러·볼륨 · 보톡스 · 레이저 · 기타.

### 7.4 경계 규칙(반드시 준수)

1. **방향은 한쪽으로만**: 마스터 → 양쪽(읽기). 내부 툴에서 급히 추가한 발음 변형(예: '하이라웨이브' 같은 표기)이 거꾸로 대중 앱에 새어나가지 않게 한다. **추가는 항상 마스터에서.**
2. **내부 전용 정보는 섞지 않는다**: 원장만 아는 약칭·발음 습관 메모는 `asr_aliases`(내부)에만 둔다. 마스터엔 **공식 표기만** 적재한다.

### 7.5 사전 운영 방식(선택지)

| 방식 | 장점 | 비고 |
|---|---|---|
| Supabase 테이블 + 추가 화면 | 기존 인프라 활용, 생태계 내 관리 | **권장**(이미 Supabase 사용) |
| 구글시트 1장 | 가장 쉬움, 비개발자 편집 | 소규모 시작에 적합 |
| 각 PC 로컬 파일 | 서버 0 | 공유 안 됨 → 비권장 |

## 11. 개인정보·보안·컴플라이언스

피부과 특성상 환자 사진·동의서를 다루므로, 개인정보 보호가 본 제품의 핵심 가치다.

1. **무전송 원칙**: 트랙 1·2 모두 파일이 처리 서버로 전송되지 않는다. 트랙 1은 브라우저 내, 트랙 2는 로컬 PC 내에서만 처리된다.
2. **비식별화 도구 내장**: EXIF 제거·모자이크/블러로 외부 공유 전 개인정보를 정리한다.
3. **공유 데이터 최소화**: 외부와 공유되는 것은 환자 정보가 아니라 **시술명 텍스트 데이터**뿐이다.
4. **저장하지 않음**: 결과물을 서버에 보관·적재하지 않고 즉시 다운로드한다.
5. **릴스 소재 유의**: 환자 전/후 사진을 릴스에 사용할 때는 동의 범위를 확인하고, 필요 시 모자이크·비식별화를 선행한다.

## 12. 제약 및 한계

1. **브라우저 연산 천장**(트랙 1): 메모리 한계로 대용량(특히 영상)은 부적합. → 트랙 2로 분리해 해소.
2. **AI 품질 격차**: 누끼·업스케일 품질이 전용 유료 서비스보다 떨어질 수 있다.
3. **최초 로딩 부담**: WASM·AI 모델 첫 다운로드(누끼 40~80MB, ffmpeg.wasm 약 25MB)가 무겁다.
4. **자막 정확도**: 전문용어 오인식 → 사람 검수 1회 필수.
5. **PDF 변환 한계**: PDF→워드/엑셀은 범위 밖.
6. **설치 부담**(트랙 2): 각 PC 최초 1회 설치 필요(이후엔 아이콘 실행).
7. **유지보수**: 라이브러리 업데이트·버그 대응에 개발자 손이 필요.
8. **릴스 템플릿 선행 설계**: 흐름을 먼저 고정 설계해야 변수 교체가 가능하다. Remotion 채택 시 상업 라이선스 확인이 필요하다.

## 13. 개발 로드맵 및 우선순위

| 단계 | 범위 | 목표 | 비고 |
|---|---|---|---|
| **Phase 1** | 트랙 1 핵심 묶음 | QR · 누끼 · PDF · 비포애프터 | 즉시 사용, 효과 대비 난이도 낮음 |
| **Phase 2** | 트랙 1 확장 | 압축/WebP·HEIC·리사이즈·EXIF·모자이크·워터마크·GIF | 개인정보·SNS 워크플로우 강화 |
| **Phase 3** | 공유 사전 구축 | `procedures_master` + `asr_aliases` | 자막 정확도의 토대 |
| **Phase 4** | 트랙 2 기반 + 자막 | Whisper + LLM 교정 + ffmpeg, 각 PC 설치 | 로컬 도입 시작 |
| **Phase 5** | 릴스 템플릿 자동화 | 전/후·나이·성별·코멘트 변수 + 배치 생성 | **고가치**. 샘플 릴스 확보 시 앞당겨 진행 가능 |
| **Phase 6** | 영상 업스케일·대용량 변환 | Real-ESRGAN / Video2X (필요시 Topaz) | GPU 풀 활용 |

**권장 시작점**: Phase 1의 **QR 생성기** 또는 **비포·애프터 슬라이더**(난이도 낮고 효과 큼). 릴스(Phase 5)는 트랙 2 기반(Phase 4)과 ffmpeg 합성을 공유하므로, 실제 샘플 릴스가 준비되면 우선순위를 올릴 수 있다.

### 15.2 용어집

- **ASR**: 음성인식(Automatic Speech Recognition). 소리→글자.
- **Whisper**: OpenAI가 공개한 무료 ASR 모델(LLM 아님).
- **WASM**: WebAssembly. 브라우저에서 네이티브 수준 연산을 가능케 하는 기술.
- **COOP/COEP**: `SharedArrayBuffer` 사용을 위한 보안 헤더. ffmpeg.wasm·AI 모델 동작에 필요.
- **하드섭/소프트섭**: 자막을 영상에 굽기(hard) / 별도 트랙 첨부(soft).
- **Single Source of Truth**: 데이터 원본을 한곳에 두어 중복·불일치를 막는 원칙.
- **변수 슬롯**: 템플릿에서 매번 바뀌는 값을 끼우는 자리(릴스의 전/후 사진·나이·코멘트 등).
- **Remotion**: 코드(React)로 영상 템플릿을 만들어 변수로 렌더하는 도구.
